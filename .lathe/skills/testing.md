# Testing in gosq-codegen

This skill exists to answer: how does this project test, what conventions do existing tests follow, and what should new tests look like?

---

## Two tiers, kept separate

**Unit tests** — `go test ./...` with no flags. No database. No network. Must work on a fresh clone with only Go installed. These construct `[]introspect.Table` values inline and assert codegen output.

**Integration tests** — `go test -tags integration ./...`. Require a real PostgreSQL instance. Skip automatically when `TEST_DSN` is not set. Run in CI via a Postgres 16 service container.

Never mix the tiers. A test that calls `sql.Open` belongs behind `//go:build integration`.

---

## Unit test conventions (codegen_test.go)

All unit tests are in `package codegen` (internal package test — can access unexported functions like `toExported`).

**Exact string assertions:** Most tests construct `[]introspect.Table` inline and compare `string(got)` to a literal `want` string. This documents the exact expected output character by character, including `go/format`'s tab-aligned `=` signs in `var` blocks. When writing a want string for a multi-column table, get the actual output first — don't guess the alignment.

```go
got, err := Generate(tables, Config{Package: "schema", DotImport: true})
if err != nil {
    t.Fatalf("unexpected error: %v", err)
}
want := "// Code generated by gosq-codegen; DO NOT EDIT.\n\npackage schema\n\n..."
if string(got) != want {
    t.Errorf("output mismatch\ngot:\n%s\nwant:\n%s", got, want)
}
```

**Error-only tests:** Collision tests only check that an error is returned — not the message string. This lets the wording change without breaking tests.

```go
_, err := Generate(tables, Config{Package: "schema", DotImport: true})
if err == nil {
    t.Fatal("expected error for colliding table identifiers, got nil")
}
```

**Table-driven subtests:** `TestToExported` uses a `[]struct{ in, want string }` table with `t.Run(tc.in, ...)`. Use this pattern for `toExported` test additions.

**Compile verification test:** `TestGenerateCompiles` writes generated output into a `t.TempDir()` module with a minimal gosq stub and runs `go build ./schema`. The gosq stub pattern:
```go
write(filepath.Join(dir, "gosqstub", "go.mod"), "module github.com/libliflin/gosq\n\ngo 1.22\n")
write(filepath.Join(dir, "gosqstub", "gosq.go"), `package gosq
type Table struct{}
type Field struct{}
func NewTable(name string) Table { return Table{} }
func NewField(name string) Field { return Field{} }
`)
// root go.mod: replace github.com/libliflin/gosq => ./gosqstub
```

**Production-scale test:** `TestGenerateProductionScale` (17 tables, 108 columns) uses `strings.Contains` for identifier names and field arguments rather than exact output comparison — correct because `go/format` alignment is fragile for large outputs.

## Integration test conventions (integration_test.go)

All integration tests are in `package introspect_test` (external test package — avoids the `codegen` → `introspect` import cycle).

**Build tag:** `//go:build integration` on the first line (before the package declaration).

**DB helper pattern:**
```go
func openIntegrationDB(t *testing.T) *sql.DB {
    t.Helper()
    dsn := os.Getenv("TEST_DSN")
    if dsn == "" {
        t.Skip("TEST_DSN not set; skipping integration test")
    }
    db, err := sql.Open("postgres", dsn)
    // ... ping, t.Cleanup
}
```

**Schema isolation:** Every test creates a unique schema name, drops-if-exists at the start, drops-if-exists in `t.Cleanup`. DDL fixtures are loaded via a dedicated connection (`db.Conn`) with `SET search_path`:
```go
conn, err := db.Conn(ctx)
conn.ExecContext(ctx, "SET search_path TO "+schema)
conn.ExecContext(ctx, string(ddl))
```
This is required because DDL in fixtures (e.g., `CREATE TABLE users ...`) is schema-relative. The `SET search_path` must apply to the same connection that executes the DDL.

## DDL fixtures in testdata/schemas/

| File | Tables | Purpose |
|------|--------|---------|
| `ecommerce.sql` | 2 (users, orders) | Basic integration test: NOT NULL and nullable columns |
| `non_ascii.sql` | 1 (articles) | Non-ASCII column names (éditeur, prénom) |
| `reporting.sql` | 1 (reports) | Schema isolation test — coexists with ecommerce in a separate schema |

All fixtures are minimal — the smallest DDL that exercises the pattern under test. New fixtures: a comment header, then plain `CREATE TABLE` statements. No psql meta-commands.

**Gap:** All integration fixtures are small (1–2 tables). The 17-table/108-column synthetic data in `TestGenerateProductionScale` has no integration-tier equivalent. A larger DDL fixture would exercise the full pipeline with realistic schema complexity.

**Gap:** No fixture contains views. The `WHERE t.table_type = 'BASE TABLE'` filter in `introspect.Tables` excludes views, but this has never been verified against a schema that actually has views.

## What new tests should look like

**New codegen unit tests:** Add to `codegen_test.go` in `package codegen`. Use exact string comparison for small outputs, `strings.Contains` checks for large outputs.

**New integration tests:** Add to `internal/introspect/integration_test.go` in `package introspect_test`. Follow the schema-isolation pattern. Load a fixture from `testdata/schemas/` or use inline DDL for simple cases.

**New DDL fixtures:** Place in `testdata/schemas/`, add a comment header, keep it minimal. Name them after what they test, not after what they contain (`views.sql`, not `blog_schema.sql`).

**The uncovered path:** The `format.Source` error return in `Generate` is unreachable from valid input. Don't try to test it.

## Running tests

```bash
go test ./...                      # Unit tests (no DB required)
go test ./... -v                   # Verbose
go test ./... -run TestGen         # Matching tests only
go test ./... -count=1             # Disable cache

# Integration tests (requires Postgres):
docker run --rm -p 5432:5432 -e POSTGRES_PASSWORD=test postgres:16
TEST_DSN="postgres://postgres:test@localhost:5432/postgres?sslmode=disable" \
  go test -tags integration ./...
```
