# Code Quality in gosq-codegen

This skill exists to answer: what does "quality" mean specifically for this project beyond standard Go idioms?

---

## Generated code must be gofmt-clean

`codegen.Generate` returns `[]byte` that will be committed to users' version control. It must:
1. Compile without errors in a real Go module that imports gosq
2. Be identical to what `gofmt` would produce

`Generate` pipes output through `go/format` before returning. Never remove this step.

---

## Determinism is not optional

Generated files live in version control. Same `[]introspect.Table` + same `Config` → identical `[]byte`, always. Tables are sorted alphabetically before rendering. Columns rendered in `OrdinalPos` order. No map iteration in the rendering path. This is already correct — don't change it.

---

## Identifier naming: stability beats perfection

`toExported` is correct, well-tested (43 subtests), and stable. Changing it renames variables in users' codebases.

**Don't change `toExported`** for aesthetics, theoretical correctness, or speculative inputs.

**Do change `toExported`** if a user reports a real column name produces a wrong or non-compilable identifier. Fix it, test it, document it as a naming change.

The current initialism list (17 entries): `id`, `url`, `uri`, `http`, `https`, `sql`, `api`, `uid`, `uuid`, `ip`, `io`, `cpu`, `xml`, `json`, `rpc`, `tls`, `ttl`. Adding an initialism is a breaking change — it renames variables for users with those column names.

---

## Error messages must name what failed

```go
// Good
return nil, fmt.Errorf("tables %q and %q both produce identifier %q", prev, tbl.Name, ident)
return nil, fmt.Errorf("table %q produces blank identifier %q; it cannot be referenced in Go", tbl.Name, ident)

// Not helpful
return nil, fmt.Errorf("identifier collision")
```

`main.go` prefixes errors with `"gosq-codegen: "` and the operation name. Internal packages don't include their own package name — callers add context. This is already correct; maintain the pattern.

---

## Keep `internal/` internal

Only export what `main.go` or tests actually use:
- `introspect.Table`, `introspect.Column` — used by codegen
- `introspect.Tables` — called by main
- `codegen.Config` — passed by main
- `codegen.Generate` — called by main

Don't add exported symbols without a concrete caller.

---

## Static analysis

Before marking a cycle complete:
```bash
go build ./...    # Must succeed
go test ./...     # Must pass
go vet ./...      # Must be clean
staticcheck ./... # Must be clean (v0.5.1 pinned in CI)
```

Don't suppress warnings with `//nolint` unless the warning is a provably false positive — document why.

---

## The `// Code generated` header

Every generated file begins with:
```
// Code generated by gosq-codegen; DO NOT EDIT.
```

This is the Go convention. Linters, editors, and code review tools use it to suppress warnings for machine-generated files. Never remove it from `Generate`.

---

## CI quality

`.github/workflows/ci.yml` runs on every push and PR:
1. `go build ./...`
2. `go vet ./...`
3. `go test ./...` — unit tests
4. `go test -tags integration ./...` — integration tests against Postgres 16 service
5. `staticcheck ./...` — pinned at v0.5.1

**Known CI weakness:** `staticcheck` is installed via `go install @v0.5.1` on every run — not cached, adds ~20–30 seconds. Could be replaced with a pre-built binary or `staticcheck-action` that benefits from caching.

**Actions currency:** `actions/checkout@v6.0.2` and `actions/setup-go@v6.4.0` — both use Node.js 22 as of April 2026. Monitor for newer releases.
